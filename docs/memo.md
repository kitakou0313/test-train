# 用語の定義
ISTQBにおける定義。SWEBOKなどの定義とは異なるので注意

- エラー
    - 誤った結果を生み出す人間の行為
- フォールト（欠陥、バグ）
    - 要求された機能を果たせなくするソフトウェア、システム内の不備
- 故障（failure）
    - ソフトウェア、システムが要求された機能を提供できないこと
- インシデント
    - テストの結果、期待動作と得られる結果が異なる状態

エラー -> フォールト       -> 故障

# テストの原則
- テストは「欠陥があること」しか示せない
    - テストに成功した -> そのソフトウェアに欠陥がない は必ずしも成り立たない
        - 失敗ケースを未テストだった、たまたま実行時に故障しなかった など
    - テストで考慮したケースにおいてそのテスト実行時に欠陥があったことのみを示せる
- 全数テストは不可能
    - 有限の資源（時間など）において全てのケースを考慮するのは困難
        - 正常系、異常系....
    - テストに優先順位をつけて実施する
- 開発初期からテストする
    - ソフトウェアの修正は後段になる程困難になる
    - 開発の初期からテストすべき
- 欠陥の偏在
    - テストはソフトウェアの全ての部分に均等に存在しているわけではない
    - 偏在している
        - これに基づいてテストの絞り込みを行うべき
- 殺虫剤のパラドックス
    - 同じテストを繰り返していると、そのテストでは欠陥を見つけられなくなる
        - 開発者が集中して修正を行うため
    - プロパティベーステスト？
- テストは条件次第
    - ソフトウェアが使われる環境によってテストを変えるべき
        - そのシチュエーションでの欠陥を発見するため
        - 全数テストが不可能なので、資源を集中するべき
- 欠陥が0 -> 機能を十全に果たすソフトウェア ではない
    - 欠陥が0にできたがレスポンスタイムが長時間 -> 使い物にならない
