# テスト分析と設計

Glenford Myers推奨のテスト設計の流れ

- 仕様が入力条件の組み合わせを含んでいる場合は原因-結果グラフの作成から始める
- どんな場合でも境界値分析を行う
- 入出力の有効無効パーティションを見分け、必要ならテストケースを補足
- エラー推測技法によりテストケースを補足
- 要求されるカバレッジに上の手順に満たない場合、ホワイトボックス技法によりテストを追加する

## テスト技法の分類

### ブラックボックス
- テストアイテムやテスト条件に対する要件が明らかになっている前提
- テスト対象の入出力にのみ注目したテスト技法
    - テストケースの網羅度に注目する

### ホワイトボックス
- テスト対象ソフトウェアの内部構造（コード、ドキュメント）が明らかになっている前提
- テスト対象のソフトウェアの内部構造や処理に注目する技法
    - C1, C0などのカバレッジに注目する

### 経験ベースのテスト技法 
- 開発者、テスト担当者に経験がある前提
- 開発者、テスト担当者の経験に基づいてテストケースを作成する
- 最終確認の役割として一定の効果が見込まれる
    - 探索的テスト


ブラックボックス、ホワイトボックスの中間としてグレーボックステスト技法もある コードの詳細よりも荒い粒度でテストする


## ブラックボックステスト技法の詳細
- 同値分割法
    - 同値パーティションごとに代表的な値を一つ選びテストする
        - 同値パーティション -> ソフトウェアへの入力, 出力データのうち同じ処理が行われると想定した集合
        - 例
            - 内部変数
            - 時間関連の値
            - インターフェースのパラメーター
            - 構成アイテム
            - 順序関係の有無、離散 or 連続
        - パーティションはMECEになるように定義する
            - 重複あり -> どのパーティションに対するテストをしたのか不明
            - 漏れあり -> テストケースのもれ
        - 有効パーティション、無効パーティションについても考慮してテストする
            - 無効パーティション -> テスト対象が拒否すべきもの、仕様で定義されていないもの
            - ソフトウェアの内部構造を意識せず、入力されるデータの集合にのみ注目すること
    - カバレッジの算出
        - テストしたパーティション数 / 全パーティション数
- 境界値分析（Boundary Value Analysis）
    - 順序関係があるデータを扱うソフトウェアのテストに用いられる手法
    - 境界値およびその隣接値をテストする手法
        - 境界値 -> 各パーティションに含まれる値のうち、下限又は上限
            - 
        - 条件の境界の値に仕様の考慮もれが集中するため、集中的にテストする手法
    - 種類
        - 前提
            - 「20歳以上, 34歳以下」の年齢のユーザーに対するテストを考える
                - このパーティションF1に属する境界値は「20」と「34」
            - 「35歳以上」を含むパーティションF2が存在している
            - 入力の全集合は20 <= x <= 100の整数とする
        - 2値VBA
            - 境界値にくわえ、各パーティションごと隣接するパーティション内で各境界値に隣接する値をテストする
            - 19, 20, 34, 35をテストする
        - 3値VBA
            - 境界値にくわえ、各パーティションごと隣接するパーティション内で各境界値に隣接する値とパーティション内の値のうち各境界値に隣接する値をテストする
            - 19, 20, 21, 33, 34, 35をテストする
                - F2の境界値を35とした場合、34, 35, 36をテストすることになり、3値VBAとしては不適切
                    - メモ:不適切だと何が良くない？
                - そのため、あくまでも境界値は34のみであり34 < x < 35に関しては仕様を整備した方が良い
    - カバレッジの算出
        - テストした境界値とその隣接値の数 / テストすべき境界値とその隣接値の個数
- デシジョンテーブルテスト
    - 入力データ、入力条件とその時の結果の組み合わせを網羅的にまとめたもの
        - 各条件が満たされているかの組み合わせとその時の結果を列挙する
        - この各列をテストケースとする
    - カバレッジの算出
        - テストを実行した列数 / 実行可能な列の総数
- 状態遷移テスト
    - 状態をもつシステムに対するテスト技法
        - 前処理が同じならば後処理も同じ
        - もし異なるのであればテスト対象のシステムが追加の情報を持っている -> 状態
    - あるシステムがとりえる状態とその状態間の遷移をDAGで表した図 -> 状態遷移図
    - これらの状態とその遷移の組み合わせをテストする技法
    - カバレッジの算出
        - 全状態カバレッジ
            - テストを実行した状態の数 / 状態の総数
        - 有効遷移カバレッジ
            - テストを実行した有効な遷移の数 / 有効な遷移の総数
        - 全遷移カバレッジ
            - テストを実行した有効および無効な遷移の総数 / 有効および無効な遷移の総数

## ホワイトボックステスト技法の詳細
- カバレッジの種類
    - ステートメントカバレッジ
        - テストを実行されたソースコードの行数 / 全ての行数
    - ブランチカバレッジ
        - ブランチの求め方 コードを以下の部分に分ける
            - 各分岐
                - 条件が真の時に実行される処理
                - 条件が偽の時に実行される処理
            - 繰り返しの中の処理
            - 上記以外の部分
                - 実行開始 -> 分岐 or 繰り返しまで
                - 分岐 or 繰り返し -> 分岐 or 繰り返しまで
                - 分岐 or 繰り返し -> 実行終了
                - 実行開始 -> 実行終了
        - テストを実行されたソフトウェアの分岐数 / 全ての分岐数
    - カバレッジ間の関係
        - ブランチカバレッジはステートメントカバレッジを内包する
            - ブランチカバレッジ100%ならばステートメントカバレッジ100%を満たす
            - 逆は成り立たない
                - 処理をスキップするようなブランチがあった場合、そのスキップのブランチがテストされないため
    - ブランチカバレッジ100% -> 欠陥がない は必ずしも成り立たない
        - 入力データの特定の値で現れる欠陥が検出されないこともある
        - 実装されていない仕様についての欠陥が検出できない
- ホワイトボックステスト技法の価値
    - 実装上の欠陥の検出
        - 分岐の境界についてのエラー、初期値の設定誤りなど
        - 未実装の仕様については検出できない
    - コードレビュー
        - ブランチカバレッジを意識してレビューすることで網羅的なテストができる
    - コードカバレッジの測定
        - カバレッジに基づいた意思決定をステークホルダーに提供できる
            - テストの追加, 集中的にテストすべきコンポーネントの特定など

## 経験ベーステスト技法
- エラー推測
    - エラー、欠陥、故障の類型をベースにテストケースを考える手法
- 探索的テスト
    - テスト担当者にテストのガイド（テストチャーター）を元に一定の時間（セッション）内でテスト対象を操作してもらい、FBを受ける手法
- チェックリストベースドテスト
    - チェックリストに記載されたテスト条件をカバーするようにテストケースを作成する手法
    - 以下の視点で作成される
        - 経験
        - ユーザーにとって重要な点
        - ソフトウェアが不合格になる点とその仕組み
    - 以下の視点は含めない
        - 一般的かつ曖昧な視点
            - 「コードは保守可能？」などの視点
            - プロジェクトに関わる特有
        - 自動的に検出可能な視点
            - 静的テストなどで発見できるもの
